A quant research harness that refuses to lie about edge.

mt5-research pairs a disciplined, out-of-sample backtesting laboratory with a demo-gated autonomous MT5 executor. Nothing trades until a strategy survives real costs on held-out data — and so far, out of 34 configurations tested, nothing has.

Two planes, one honest system

A research plane finds out whether an idea survives contact with real costs. An executor plane only ever trades what the same bar has already cleared on fresh broker data. Every skip, rejection, and closed trade is journaled by both.

Research plane
repo root · numpy + stdlib only
  • Zero LLMs anywhere in the decision loop — deterministic, rule-based strategies only.
  • backtest.py → robustness.py → walkforward.py → portfolio.py: cost audit, parameter-surface verdict, rolling out-of-sample validation, cross-asset aggregation.
  • risk.py is the only module allowed to approve a position size; execution.py hard-refuses any non-demo account before an order is even built.
  • registry.py logs every out-of-sample-touched config — the running "multiple-testing" luck budget.
Executor plane
intel/executor/ · demo-gated, 24/7
  • engine.py runs a 30-second reconcile → manage → decide → order loop.
  • bridge_server.py is the only file in the entire codebase allowed to call order_send, and it re-verifies the account is DEMO before every single write.
  • gate.py backtests every strategy×symbol on 5,000 fresh broker bars — a combo trades only after it passes out-of-sample.
  • review.py writes a post-mortem on every closed trade; store.py journals all of it to SQLite for the dashboard.

research plane finds the verdict → executor plane's gate enforces it, per strategy × symbol, on your own broker's data

Nine layers, all server-side

Live trading is off by default and structurally difficult to turn on. There is no config flag, env var, or dashboard toggle that skips any layer below — they are re-checked on every single order, not just at startup.

  1. 1Writes refused unless the account is DEMO, re-checked per order.bridge_server.py
  2. 2Orders without SL and TP are refused — no naked positions.bridge_server.py
  3. 3Volume above the cap (0.50 lots by default) refused.bridge_server.py
  4. 4Live accounts require the triple unlock below — no exceptions.bridge_server.py
  5. 5Must pass the out-of-sample backtest gate before a combo may trade.gate.py
  6. 60.5% risk sizing, 2% daily halt, 10% drawdown halt.risk.py
  7. 7News blackout ±30 minutes around high-impact events.risk.py
  8. 8Cooldown after 3 straight losses; disabled after 5 losses in 7 days.review.py
  9. 9Kill switch — touch executor/data/KILL flattens everything and halts.engine.py
The triple-lock for live: a real account only ever accepts an order when three independent things are true at once — the env flag MI_ALLOW_LIVE=1, a file containing the exact account login, and that login matching the terminal's logged-in account — re-verified before every order. Flipping one switch is not enough, on purpose.

Every out-of-sample-touched config is logged to a results registry with a running multiple-testing counter. At a 5% false-positive rate, roughly 1.7 of 34 tested configs would look like "winners" by pure chance alone — which is exactly why a Sharpe of 0.28 is reported as GATE NOT MET instead of a headline.

Quick start

Clone it, ask it what's missing, then decide how far to run it — nothing here defaults to sending real orders.

git clone https://github.com/Kalahari-Labs/mt5-research
cd mt5-research

./run.sh check    # probes every prerequisite, tells you what's missing
./run.sh gate     # backtests every strategy x symbol on YOUR broker's data
./run.sh observe  # runs the full pipeline, journals every decision, sends zero orders
./run.sh          # autonomous — still demo-gated, always

Root research suite: python3 -m unittest discover -s tests — 156 tests, numpy + stdlib only, no network calls, no broker required.
